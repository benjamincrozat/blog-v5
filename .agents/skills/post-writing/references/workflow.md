# Post Writing Workflow (Detailed)

## Draft Workflow

1. Define the reader promise in one sentence before drafting.
2. Draft at least five title candidates before selecting one.
3. Choose the title that best passes this gate: a novice understands it quickly, it promises one concrete outcome, it uses sentence case (except acronyms/official names), it is 60 characters or fewer, and it creates curiosity without hype.
4. Identify the exact reader problem, likely context, and key decision.
5. Define one primary keyword and a few close variants that match reader search intent.
6. Research primary sources first; confirm versions, dates, and claims with official sources.
7. Keep only evidence that materially improves reader decisions.
8. Build a focused outline from takeaway to essentials to implementation.
9. Draft in clear first person only where it improves trust; prioritize second-person action guidance.
10. Define unavoidable jargon in plain language at first mention, then reuse the short term naturally.
11. Add practical code examples where they materially improve clarity or implementation speed.
12. Frame every code snippet with context and explanation: what problem it solves, what the reader should notice, and how to validate results.
13. Add inline links exactly where context is needed; avoid trailing link dumps.
14. Check every in-post link to confirm it resolves and stays relevant to the nearby claim.
15. Execute draft commands/snippets when feasible to verify they work as written.
16. When correctness depends on visuals (UI, charts, layout), verify against source material and avoid browser-based validation for all post-writing tasks (new drafts and revisions).
17. Keep only images that directly support the reader task.
18. Revise for brevity by removing repetition, tangents, and self-focused narrative.
19. Remove temporary validation artifacts and process narration that does not help the reader complete the task.

## Voice And Focus

- Keep tone practical, specific, and instructional.
- Prefer short, plain sentences and common words to keep readability around 5th-grade level.
- Use first person only for informed judgment and concrete tradeoffs.
- Avoid rhetorical flourishes and vague positioning language.

## Research And Evidence

- Prefer official docs, maintainers, standards bodies, and primary announcements.
- Prefer current information for APIs, libraries, policies, and timelines.
- State uncertainty when sources conflict or evidence is incomplete.
- Avoid unsupported claims and low-value citations.

## Linking

- Place links inline at the supporting claim/action.
- Verify each link resolves and directly supports the nearby claim/context.
- Prefer internal links where coverage already exists.
- Use external links to fill gaps, not duplicate existing on-site coverage.
- Use descriptive anchor text that communicates intent/outcome.
- Prefer one strong contextual link over multiple redundant links.

## Structure

- Start with the main takeaway.
- Keep only essentials needed for reader action.
- Use short sections with concrete transitions.
- Use descriptive unnumbered headings (avoid `1)`/`2)` style).
- Use the exact primary keyword in only 1-2 strategic H2/H3 headings (usually one early section and one high-intent section).
- Use close variants or direct topic labels in other headings, and avoid repeating the same keyword prefix across consecutive headings.
- Read headings top-to-bottom as a table-of-contents test; if they sound repetitive or pushy, rewrite for natural language first.
- End with practical next steps, caveats, or verification checks when relevant.
